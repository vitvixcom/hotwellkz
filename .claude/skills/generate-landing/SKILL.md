# Генерация SEO-лендинга HotWell.kz (полная копия главной)

## Что делает
`generate.py` берёт `site/index.html` целиком (ВСЕ блоки и скрипты) и адаптирует
под `[город] + [услугу]` только то, что важно для SEO и релевантности:
1. `<title>`, `meta description`, `meta keywords`, `canonical`.
2. Open Graph и Twitter (`og:title/description/url`, `twitter:*`).
3. **Hero**: надзаголовок (eyebrow), `<h1>` (с выделением услуги), лид-абзац — под
   город и услугу (пишет OpenAI как продающий копирайтер).
4. Добавляет JSON-LD `Service` (с `areaServed = город`) + `BreadcrumbList`
   (рядом с уже существующей разметкой Organization/LocalBusiness/FAQPage главной).
5. В форму заявки добавляет скрытое поле `Источник = "Лендинг: <город> / <услуга>"`.

Всё остальное (дизайн, калькулятор, отзывы, скорость, адаптивность) наследуется
от главной без изменений.

## Куда кладётся
Страница пишется в КОРЕНЬ сайта — `site/<slug>.html` — поэтому все относительные
пути (css, js, картинки, ссылки) работают как на главной, без переписывания.
URL: `https://hotwellkz.kz/<slug>.html`. Страница добавляется в `sitemap.xml`.

## Требования
- `OPENAI_API_KEY` (env или файл `.openai_key` рядом со скриптом / в соседнем
  скилле `generate-article`). Без ключа или с `--no-ai` берутся качественные
  шаблонные заголовки.
- `site/index.html` (актуальная главная) и общий `css/style.css` — уже в репозитории.
  Важно: лендинг = снимок главной на момент генерации. Если главную обновили и нужно
  подтянуть изменения в лендинг — перегенерируйте его.

## Как запускать (шаги для агента)
1. Из корня репозитория:
   ```bash
   python3 .claude/skills/generate-landing/generate.py --city "Астана" --service "каркасные дома"
   python3 .claude/skills/generate-landing/generate.py --city "Шымкент" --service "дома из СИП-панелей" --kw "цена, под ключ"
   ```
   Флаги: `--kw "…"` доп. ключевые слова; `--no-ai` без OpenAI.
2. Проверить: число `<section>` совпадает с главной (все блоки на месте), JSON-LD
   валиден, обновились title/H1/canonical.
3. Закоммитить (`site/<slug>.html`, `site/sitemap.xml`, `published.json`) и
   **задеплоить на Netlify** (форма заявки заработает после деплоя).

## Синхронизация с главной (ВАЖНО — единый источник правды)
Главная `site/index.html` — это МАСТЕР-ШАБЛОН для всех лендингов. У каждого лендинга
в `published.json` хранятся только его SEO-данные (title, description, keywords, H1,
eyebrow, lead). Поэтому при изменении любого блока на главной правки разносятся по
ВСЕМ лендингам одной командой, без обращения к OpenAI и без ручного редактирования
каждой страницы (масштабируется на сотни страниц):
```bash
python3 .claude/skills/generate-landing/generate.py --sync
```
`--sync` берёт текущую `index.html` и пересобирает каждый лендинг из неё, заново
подставляя сохранённый SEO каждого города. **Рабочий процесс:** поменяли блок на
главной → запустили `--sync` → задеплоили. Все города обновились.

(Если у лендинга в `published.json` нет сохранённого SEO — он будет пропущен при
`--sync`; такие старые страницы нужно один раз перегенерировать обычной командой.)

## Внутренняя SEO-панель (site/seo-panel.html)
Скилл строит и обновляет внутреннюю страницу `site/seo-panel.html` — чтобы не лазить
в репозиторий. На ней:
- **Таблица ключевых слов** из `Лист2` (объём запросов/мес — цветные бейджи; конкуренция
  цветом; цена клика USD; колонка «Страница» — под какие ключи уже сделаны лендинги
  и в каких городах). Поиск и сортировка по колонкам.
  Колонка «Страница» проставляется по **категориям услуг** (корням), а не по точному
  совпадению строки: ключевик и страница относятся к одной категории, если совпадает
  хотя бы один корень из `CATEGORY_RULES` в `generate.py` (каркас, сип, сэндвич,
  модульн, баня, гараж, дачa…). Поэтому «каркасный дом», «каркасные дома под ключ»,
  «строительство каркасных домов» и т.п. помечаются всеми городами, где есть каркасная
  страница; аналогично для СИП. **Новую услугу → добавьте корень одной строкой в
  `CATEGORY_RULES`.**
- **Сводка «Интенты услуг»** (сверху): каждая категория одной строкой — сколько ключей,
  суммарный объём/мес, покрытие (города со страницей либо «нет страницы» красным). Это
  карта каннибализации и незакрытых возможностей: один интент = одна страница на город;
  если у интента большой объём и «нет страницы» — туда стоит сделать лендинг.
- **Колонка «Интент»** в таблице ключей: показывает, к какой странице относится ключ
  (по `categories_of`). Долгое удержание ключа копирует его в буфер обмена.
- **Таблица созданных страниц** по городам: город → ключ/услуга → H1 → ссылка → дата.

Под каждый синоним одного интента отдельную страницу делать НЕ нужно — это
каннибализация; хватает одной страницы на город. Колонка «Страница»/«Интент» —
индикатор: если ключ уже «✓ <город>», интент закрыт.

Страница **закрыта от индексации** (`<meta robots noindex,nofollow>`), не в sitemap,
ни на что не ссылается — доступ только по прямому URL `…/seo-panel.html`.
Пересобирается автоматически при создании лендинга и при `--sync`; отдельно — командой:
```bash
python3 .claude/skills/generate-landing/generate.py --dashboard
```

## Реестр ключевиков по городам (Ключевые слова.xlsx)
При создании лендинга скилл автоматически ведёт учёт в `Ключевые слова.xlsx` (в корне
репозитория): для города создаётся отдельный ЛИСТ (если его ещё нет), и в него
записывается строка «**ключевое слово/услуга → Заголовок (H1) → Ссылка → Дата**».
Так в одном файле видно, под какой запрос какая страница уже сделана, по каждому городу.
Новый город → новый лист появляется сам. (Нужен `openpyxl`; если его нет — генерация
страницы не прерывается, просто xlsx не обновится, о чём будет сообщение.)
Исходные листы с семантикой (`Лист1`, `Лист2`) не трогаются.

## Заметки
- Заявки помечаются полем `Источник` — в Netlify → Forms видно, с какого лендинга лид.
- `published.json` хранит SEO-данные каждого лендинга (для `--sync`) + город/услуга/URL.
- Дубликатов контента бояться не нужно: canonical у каждого лендинга — собственный,
  title/H1/мета/лид уникальны под город — это стандартные локальные посадочные.
